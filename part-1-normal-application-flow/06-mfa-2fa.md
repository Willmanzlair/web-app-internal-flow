# MFA / 2FA: What Happens Internally

![MFA / 2FA Internal Flow](../images/mfa%20flow.png)

MFA, or multi-factor authentication, is the part of the authentication flow where the application requires more than a password before treating the user as fully authenticated.

A password proves something the user knows. MFA adds another proof, usually something the user has, such as an authenticator app, hardware key, email code, SMS code, or backup code.

In many modern applications, MFA does not replace the normal login flow. It extends it.

The backend may first check the user’s email and password. If the password is correct and MFA is not enabled, the backend may complete login normally. But if MFA is enabled for that account, the backend should not immediately issue full authenticated access after the password step. Instead, it should pause the login flow and require the second factor.

This creates an important distinction:

```text
Password accepted does not always mean full login completed.
```

When MFA is enabled, the password may only complete the first part of authentication. The second factor must still be verified before the backend issues the final authenticated state.

A clean mental model is:

```text
Password = first proof
MFA code = second proof
Pending MFA state = password passed, second factor still required
Access token/session = full authenticated state
```

MFA is therefore not just another screen in the interface. It is a second trust decision controlled by the backend.

---

## 1. What this functionality is

MFA is a security control that requires the user to prove their identity with more than one factor.

A normal password-based login usually asks:

```text
Do you know the correct password for this account?
```

MFA adds another question:

```text
Can you also prove possession of the second factor?
```

That second factor may come from an authenticator app, email, SMS, push notification, hardware security key, or backup code.

In the flow observed here, the application uses TOTP, which means time-based one-time password.

A TOTP code is the 6-digit code generated by an authenticator app. The code changes after a short time window, commonly every 30 seconds.

The important detail is that the authenticator app is not asking the backend for a new code every time. Instead, both the backend and the authenticator app share the same secret. The authenticator app uses that secret and the current time to generate the current 6-digit code. The backend uses the same secret and the current time to calculate what the valid code should be.

If both sides calculate the same code, the backend accepts the MFA challenge.

A simple way to understand TOTP is:

```text
Backend stores MFA secret.
Authenticator app stores the same MFA secret.
Both use the secret and current time.
Both generate the same short-lived 6-digit code.
```

This is why the MFA secret is extremely sensitive. If an attacker steals the MFA secret, they can generate valid MFA codes for that account as long as the secret remains active. The attacker may still need the password or another valid login step, but the second-factor protection has been weakened because they can now reproduce the MFA codes.

MFA therefore has two major parts:

```text
MFA enrollment = setting up the second factor
MFA challenge = proving the second factor during login
```

Enrollment is where the user connects an authenticator app or second-factor method to their account.

Challenge is where the user proves that second factor during authentication.

---

## 2. What the user sees

From the user’s point of view, MFA usually appears in two places.

The first place is the account security or profile settings area. The user is already logged in, visits their security settings, and clicks something like:

```text
Enable MFA
Enable 2FA
Set up authenticator app
```

At this stage, the user is not logging in again. They are already authenticated and are changing a security setting on their account.

The application may then show a QR code, a setup key, or instructions to add the account to an authenticator app. The QR code is usually just a visual form of an `otpauth://` setup URL. That setup URL contains the information the authenticator app needs, including the account label, issuer, number of digits, code period, and most importantly, the MFA secret.

After adding the account to the authenticator app, the user sees a 6-digit code. They enter that code back into the web application to confirm setup.

This confirmation step matters. Generating a secret is not enough. The backend needs proof that the user successfully added the secret to an authenticator app and can generate a valid code from it.

The second place the user sees MFA is during a future login.

After MFA has been enabled, the user enters their email and password as usual. If the password is correct, the application may not send them directly to the dashboard. Instead, it shows an MFA challenge screen and asks for the current 6-digit code.

From the user’s point of view, the flow may feel like:

```text
Enter password
Submit login form
Enter MFA code
Access application
```

But internally, the password and MFA code are two separate proofs.

The password proves the user knows the account secret.

The MFA code proves the user has access to the second factor.

---

## 3. What JavaScript does

JavaScript acts as the behavioural layer between the visible MFA interface and the backend API.

When the user clicks “Enable MFA,” frontend JavaScript reacts to that click and asks the browser to send a request to the backend. The browser provides the environment where JavaScript runs, and it also provides the networking system that actually sends the HTTP request.

So the precise model is:

```text
User clicks the button.
Frontend JavaScript handles the click.
JavaScript prepares the API request.
The browser sends the HTTP request.
The backend receives and validates it.
```

This matters because saying “the browser sent the request” and saying “frontend JavaScript triggered the request” are both describing different parts of the same action.

JavaScript decides what request should be made. The browser transports it.

During MFA enrollment, JavaScript may send a request to start setup, receive the MFA setup details, render a QR code or setup key, collect the 6-digit confirmation code, and submit that code back to the backend.

During MFA challenge, JavaScript may collect the MFA code from the user, include a temporary pending token or challenge ID if the backend issued one, submit the challenge request, and then react to the backend’s final response.

JavaScript should not decide whether MFA is valid. It can collect the code and send it, but the backend must verify it.

A frontend check such as “the code is 6 digits” may be useful for user experience, but it is not a security decision. The real question is not only whether the code looks correct. The real question is whether the backend can verify that the submitted code matches the expected code for that user’s MFA secret at that time.

---

## 4. The API requests

MFA usually involves more than one API request.

The first request starts MFA enrollment.

A common endpoint may look like:

```http
POST /api/auth/mfa/enroll
```

This request usually happens while the user is already logged in. The backend uses the current authenticated user context to know which account is setting up MFA.

This request means:

```text
“I am the currently authenticated user, and I want to start setting up MFA for my account.”
```

The backend may respond with an MFA secret, an `otpauthUrl`, label, digit length, and period.

A response may look like:

```json
{
  "secret": "SNNYXV6BOCPNB7OQKDFR2V26K2UCYMVG",
  "otpauthUrl": "otpauth://totp/ProjectHub:manager@projecthub.dev?secret=SNNYXV6BOCPNB7OQKDFR2V26K2UCYMVG&issuer=ProjectHub",
  "label": "ProjectHub:manager@projecthub.dev",
  "digits": 6,
  "period": 30
}
```

The `secret` is the shared seed used to generate future MFA codes.

The `otpauthUrl` is the setup URL that authenticator apps understand.

The `label` is how the account may appear inside the authenticator app.

The `digits` value tells the app how many digits the code should contain.

The `period` value tells the app how long each generated code is valid before it changes.

After the user adds the secret to an authenticator app, the next request confirms enrollment.

A common endpoint may look like:

```http
POST /api/auth/mfa/verify-enroll
```

The request body may contain:

```json
{
  "code": "035244"
}
```

This request means:

```text
“I have added the MFA secret to my authenticator app. Here is the current code. Confirm that the setup works.”
```

The backend then checks whether the submitted code matches the expected TOTP code for the user’s secret.

If the code is valid, the backend can mark MFA as enabled.

Later, when the user logs in again, the normal login request may return an MFA requirement instead of issuing full authenticated access immediately.

A response may look like:

```json
{
  "mfa_required": true,
  "pendingToken": "-x9-1hQzZDWV_R-gWIriMra6li9s",
  "message": "MFA verification required. POST the OTP to /api/auth/mfa/challenge.",
  "ttlSec": 300
}
```

This means the password step passed, but the user has not completed all authentication requirements yet.

The final MFA challenge request may look like:

```http
POST /api/auth/mfa/challenge
```

With a body such as:

```json
{
  "pendingToken": "-x9-1hQzZDWV_R-gWIriMra6li9s",
  "code": "981940"
}
```

The `pendingToken` links the MFA challenge to a recent successful password step.

The `code` proves possession of the second factor.

Only after both values are valid should the backend complete authentication.

---

## 5. What the backend validates

The backend is responsible for every important MFA decision.

During enrollment, the backend first needs to confirm that the request is coming from an authenticated user. MFA setup should not be an anonymous action because the backend needs to know which account is enabling MFA.

The backend may then generate a new MFA secret for that user. This secret is the long-term value used to generate future TOTP codes.

At this stage, the backend should not treat MFA as fully enabled just because the secret was generated. The user still needs to prove that they can generate a valid code from that secret.

During enrollment confirmation, the backend receives the submitted 6-digit code. It uses the stored secret and the current time to calculate the expected TOTP code. If the submitted code matches what the backend expects, the backend can mark MFA as enabled for the account.

The important state change is:

```text
mfaEnabled: false
```

becomes:

```text
mfaEnabled: true
```

That state change should happen only after successful confirmation.

During login challenge, the backend validates a different situation. The user has already submitted a correct password, but because MFA is enabled, the backend has not completed full authentication yet.

The backend may issue a temporary pending token. This pending token is not a normal access token. It should not grant access to protected application functionality.

The pending token only means:

```text
This user passed the password step and now needs to complete MFA.
```

When the user submits the MFA challenge, the backend should validate both parts:

```text
Is the pending token valid?
Is the MFA code valid?
```

For the pending token, the backend should check whether it belongs to a recent password step, whether it belongs to the correct user, whether it has expired, and whether it has already been used.

For the MFA code, the backend should check whether the code matches the expected TOTP code for that user’s MFA secret within the allowed time window.

Only if both checks pass should the backend issue full authenticated state.

This is the key security distinction:

```text
Password accepted = partial authentication
MFA accepted = completed authentication
Session/token issued = authenticated state
```

Partial authentication means the backend has accepted one part of the proof, but not all required authentication checks have been completed yet.

## 6. What the database stores or retrieves

The database stores the facts that allow the backend to manage MFA.

For a TOTP-based MFA flow, the database may store values such as:

* whether MFA is enabled for the user
* the user’s MFA secret
* MFA method type
* backup codes or recovery code hashes
* pending enrollment state
* failed MFA attempt counters
* MFA lockout state
* last successful MFA verification time

The database does not verify MFA by itself. It stores the values the backend needs.

The backend retrieves those values and applies the MFA logic.

A clean model is:

```text
Database stores MFA facts.
Backend verifies MFA.
Frontend displays MFA state.
```

The MFA secret is the most sensitive stored value in this flow. If an attacker steals the MFA secret, they can generate valid time-based codes for that user as long as the secret remains active.

That is why production applications should protect MFA secrets carefully. They should not casually expose the secret after setup, and they should avoid returning it in places where it is not needed.

In a training application, the secret may be shown clearly to help learning. In a production application, the secret is usually shown only during setup and then treated as sensitive account security material.

The database may also store a simple state such as:

```json
{
  "mfaEnabled": true
}
```

That state tells the backend that future logins for this user require MFA after the password step.

## 7. What response comes back

MFA responses depend on the stage of the flow.

During enrollment, the backend may return setup material.

Example:

```json
{
  "secret": "SNNYXV6BOCPNB7OQKDFR2V26K2UCYMVG",
  "otpauthUrl": "otpauth://totp/ProjectHub:manager@projecthub.dev?secret=SNNYXV6BOCPNB7OQKDFR2V26K2UCYMVG&issuer=ProjectHub",
  "label": "ProjectHub:manager@projecthub.dev",
  "digits": 6,
  "period": 30
}
```

This response allows the frontend to show setup instructions, a QR code, or a manual setup key.

During enrollment verification, the backend may return confirmation that MFA is now enabled.

Example:

```json
{
  "mfaEnabled": true
}
```

This response tells the frontend that the account security state has changed. The UI may now show MFA as enabled and offer options such as disabling MFA or viewing backup codes.

During login, if the password is correct but MFA is required, the backend may return a pending MFA response.

Example:

```json
{
  "mfa_required": true,
  "pendingToken": "-x9-1hQzZDWV_R-gWIriMra6li9s",
  "ttlSec": 300
}
```

This response does not mean the user has full access. It means the login flow has paused between password verification and final authentication.

The `pendingToken` is a temporary MFA challenge token. It is not a normal access token. Its job is to connect the upcoming MFA code submission to the successful password step that happened just before it.

The `ttlSec` value tells the frontend how long that pending state is valid. For example, `300` means the user has about 5 minutes to complete the MFA challenge.

During successful MFA challenge, the backend may return the final authenticated response. This may include an access token, refresh token, user object, or session cookie depending on the application’s authentication design.

The full login response belongs mainly under login response and auth state, but the MFA-specific point is this:

```text
The final authenticated response should only be issued after the MFA challenge succeeds.
```

That is the boundary MFA is protecting.

---

## 8. What happens after the action succeeds

MFA success means different things depending on the stage.

If enrollment succeeds, the user’s account state changes. MFA becomes enabled for future logins. The frontend may update the profile or security page to show that MFA is active.

At this point, the user has not necessarily started a new login. They have changed the security settings of an already-authenticated account.

If the MFA challenge during login succeeds, the backend completes the authentication flow. The application may then issue the final token, session, or authenticated response.

The important thing is that MFA challenge success moves the user from a pending state into a completed authentication state.

Before MFA challenge success:

```text
Password was accepted.
MFA is still required.
The user is not fully authenticated yet.
```

After MFA challenge success:

```text
Password was accepted.
MFA was accepted.
The backend may now issue full authenticated state.
```

The frontend then reacts to the backend response. It may redirect the user, render the dashboard, update current user context, or call an identity endpoint such as `/api/auth/me` to confirm who the current authenticated user is.

That `/me` behaviour belongs mainly under the session and `/me` note, but the MFA-specific point is that `/me` may report MFA-related account state such as:

```json
{
  "mfaEnabled": true
}
```

This tells the frontend that MFA is enabled for the current user. It does not perform MFA by itself. It reports the current authenticated user state after the relevant authentication flow has already happened.

## 9. What to observe from a security/testing mindset

MFA testing is about understanding where the second factor is required, how the application represents pending authentication, and whether full authenticated state is withheld until MFA succeeds.

Useful observations include:

* What endpoint starts MFA enrollment?
* Is the user required to be authenticated before enrolling MFA?
* Does enrollment generate a TOTP secret?
* Is the secret exposed only during setup, or can it be retrieved again later?
* Does the backend require confirmation before setting `mfaEnabled: true`?
* What endpoint verifies MFA enrollment?
* What happens if an incorrect MFA code is submitted?
* What happens if the same code is reused?
* How many failed MFA attempts are allowed?
* Does the backend apply rate limiting or lockout to MFA attempts?
* During login, does the backend return `mfa_required: true` after a correct password?
* Does the backend issue a pending token or challenge ID?
* Is the pending token different from a real access token?
* Does the pending token expire?
* Can the pending token be reused?
* Can the MFA challenge be completed without a valid pending token?
* Does the backend issue full access only after MFA succeeds?
* Does the frontend show MFA state correctly after setup?
* Does `/me` report `mfaEnabled` correctly after MFA is enabled?

The main thing to look for is whether the application treats MFA as a real backend-controlled trust decision or only as a frontend screen.

The frontend may show an MFA page, but the backend must enforce MFA.

A strong MFA flow should behave like this:

```text
Password is correct.
Backend sees MFA is enabled.
Backend withholds full authenticated state.
Backend issues only a pending MFA state.
User submits pending token and MFA code.
Backend verifies both.
Backend then issues full authenticated state.
```

The key security lesson is:

```text
MFA is not just an extra page.
MFA is a second backend validation step.
```

If the backend issues full access before MFA succeeds, then the MFA screen may only be cosmetic. The important test is whether protected API access is actually blocked until the second factor is verified.

The final mental model is:

```text
MFA secret = long-term shared seed
TOTP code = short-lived proof generated from the secret
pendingToken = temporary proof that password step passed
MFA challenge = backend check of pending token plus code
full authenticated state = issued only after MFA succeeds
```
